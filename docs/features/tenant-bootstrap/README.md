# Tenant Bootstrap Feature

This document explains the public tenant bootstrap feature implemented on branch:

- `feat/tenant-bootstrap`

Commit:

- `85581ee`

## Goal

Expose a public endpoint that returns basic tenant bootstrap information by tenant slug.

This is useful for flows such as:

- branded sign-in bootstrap
- tenant-specific splash or landing setup
- future public configuration lookup

This implementation follows ADR 0004 because it is a safe public flow:

- it uses tenant slug as a public bootstrap input
- it does not perform protected tenant-owned writes
- it does not use client-provided tenant input for privileged authorization

## Endpoint

```text
GET /api/v1/public/tenants/{tenantSlug}
```

Example:

```text
GET /api/v1/public/tenants/coral-jovem-asa-norte
```

Current expected response shape:

```json
{
  "slug": "coral-jovem-asa-norte",
  "display_name": "Coral Jovem Asa Norte",
  "branding": {}
}
```

If tenant branding fields are populated later, they will appear inside `branding`.

## How The Feature Works

Flow:

1. the HTTP router receives `GET /api/v1/public/tenants/{tenantSlug}`
2. the tenant handler extracts `tenantSlug`
3. the tenant service validates and normalizes the slug
4. the Postgres repository runs explicit SQL through `sqlc`
5. the result is mapped into a tenant bootstrap model
6. the handler serializes the JSON response

## File-By-File Explanation

### [model.go](/Users/levilunique/Workspace/Go/coralhub-backend/internal/modules/tenants/model.go)

Defines the module-level data returned by the feature.

- `Bootstrap`
  - the response-oriented domain model for tenant bootstrap data
- `Branding`
  - optional tenant branding fields

Why it exists:

- keeps the service independent from `sqlc` generated structs
- keeps the module free from database-specific field types

### [repository.go](/Users/levilunique/Workspace/Go/coralhub-backend/internal/modules/tenants/repository.go)

Defines the repository contract used by the tenant service.

Method:

- `GetBootstrapBySlug`

Why it exists:

- keeps the service dependent on a focused capability
- avoids coupling the service directly to `pgx` or `sqlc`

### [service.go](/Users/levilunique/Workspace/Go/coralhub-backend/internal/modules/tenants/service.go)

Implements the application logic for tenant bootstrap retrieval.

What it does:

- trims the incoming slug
- rejects empty slugs
- delegates the read to the repository
- returns module-specific errors

Important errors:

- `ErrTenantNotFound`
- `ErrInvalidTenantSlug`

Why it exists:

- keeps validation and business intent out of the HTTP handler
- keeps the handler thin, as required by the architecture docs

### [http.go](/Users/levilunique/Workspace/Go/coralhub-backend/internal/modules/tenants/http.go)

Implements the tenant module’s public HTTP route registration and response serialization.

What it does:

- registers `GET /tenants/{tenantSlug}`
- converts service errors into HTTP status codes
- serializes the JSON response

HTTP mapping:

- `400` for missing or blank slug
- `404` for unknown tenant
- `500` for unexpected internal failure

Why it exists:

- keeps tenant-specific transport logic close to the tenant module
- avoids bloating the platform router with feature-specific details

### [tenants.sql](/Users/levilunique/Workspace/Go/coralhub-backend/db/queries/tenants.sql)

Adds explicit SQL for tenant bootstrap retrieval:

- `GetTenantBootstrapBySlug`

Query behavior:

- reads from `tenants`
- left joins `tenant_configs`
- filters by `slug`
- only returns active tenants

Why the `LEFT JOIN` matters:

- bootstrap still works even if a tenant has no config row yet
- that matches the current seeded state, where `tenant_configs` is empty

### [tenants.sql.go](/Users/levilunique/Workspace/Go/coralhub-backend/internal/store/postgres/sqlc/tenants.sql.go)

Generated by `sqlc` from the new SQL query.

What it provides:

- typed Go access to `GetTenantBootstrapBySlug`

Why it exists:

- keeps SQL explicit while still giving typed query access in Go

### [tenants_repository.go](/Users/levilunique/Workspace/Go/coralhub-backend/internal/store/postgres/tenants_repository.go)

Implements the tenant repository using PostgreSQL and `sqlc`.

What it does:

- calls `GetTenantBootstrapBySlug`
- translates `pgx.ErrNoRows` into `tenants.ErrTenantNotFound`
- maps database results into the module model

Why it exists:

- keeps SQL/storage specifics in the Postgres layer
- keeps the tenant service clean and focused

### [mapping.go](/Users/levilunique/Workspace/Go/coralhub-backend/internal/store/postgres/mapping.go)

Adds a small helper for translating `pgtype.Text` into `*string`.

Why it exists:

- `sqlc` generated types use `pgtype.Text` for nullable text columns

- this keeps null handling explicit and reusable

## Current Behavior With Seed Data

The initial migration seeded:

- slug: `coral-jovem-asa-norte`
- display name: `Coral Jovem Asa Norte`

There is currently no seeded `tenant_configs` row.

So the endpoint currently returns:

- tenant identity fields
- empty branding object

That is expected.

## What This Feature Does Not Do Yet

It does not:

- authenticate users
- resolve tenant from domain automatically
- create or update tenant data
- expose choir or user data
- enforce membership or role checks

This is intentionally a safe public bootstrap read flow only.
