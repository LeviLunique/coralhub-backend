# Repository Bootstrap - CoralHub

## 1. For Whom This Guide Exists

This guide is for the **human owner or maintainer of the CoralHub platform**, not for the implementation AI.

Reason:

- repository creation and remote configuration are operational ownership tasks
- they depend on your GitHub account, organization, visibility choice, and access policy
- branch protection, secrets, and repository settings should be explicitly controlled by you

The AI may help generate files and commands, but repository creation and permission-sensitive setup should be considered a human-controlled step.

## 3. Local Machine Setup for macOS ARM with Homebrew

This section is for your machine bootstrap before creating the repository.

Your current machine state is:

- `darwin/arm64`
- `go version go1.23.2`

That is workable for older projects, but for this project you should upgrade to a more current Go toolchain before starting.

### 3.1. Install Apple Command Line Tools

If not already installed:

```bash
xcode-select --install
```

### 3.2. Update Homebrew

```bash
brew update
brew upgrade
```

### 3.3. Install Base Tooling

Recommended base packages:

```bash
brew install git gh go colima docker jq sqlc golangci-lint staticcheck awscli
brew install hashicorp/tap/terraform
```

Notes:

- `colima + docker` is a good fit on macOS ARM if you do not want Docker Desktop
- `sqlc`, `golangci-lint`, and `staticcheck` should exist locally from the start
- `awscli` and `terraform` are needed when infra work begins

### 3.4. Configure Shell Paths

Make sure `Homebrew` and `Go` binaries are in your shell path:

```bash
echo 'export PATH="/opt/homebrew/bin:$PATH"' >> ~/.zshrc
echo 'export GOPATH="$HOME/go"' >> ~/.zshrc
echo 'export PATH="$PATH:$GOPATH/bin"' >> ~/.zshrc
source ~/.zshrc
```

### 3.5. Install Go-Based Extra Tools

Recommended:

```bash
go install golang.org/x/vuln/cmd/govulncheck@latest
```

Optional now, useful later:

```bash
go install github.com/golang-migrate/migrate/v4/cmd/migrate@latest
```

### 3.6. Start the Local Container Runtime

If using Colima:

```bash
colima start --cpu 4 --memory 8 --disk 60
docker ps
```

If `docker ps` works, your local container runtime is ready.

### 3.7. Verify the Toolchain

Run:

```bash
git --version
gh --version
go version
docker version
colima status
sqlc version
golangci-lint version
staticcheck -version
govulncheck -version
terraform version
aws --version
```

Expected outcome:

- all commands exist
- Go is no longer `1.23.2`
- Docker works through Colima

### 3.8. Recommended Local Project Workspace

Create a predictable workspace:

```bash
mkdir -p ~/Workspace/Go
```

Keep the project under:

```text
~/Workspace/Go/coralhub-backend
```

## 4. Local Repository Bootstrap

Inside the target parent directory, create the project folder:

```bash
mkdir -p ~/Workspace/Go/coralhub-backend
cd ~/Workspace/Go/coralhub-backend
```

Initialize Git:

```bash
git init
```

Set the default branch to `main`:

```bash
git branch -M main
```

Create the first baseline files:

- `README.md`
- `.gitignore`
- `.editorconfig`
- `docs/`

Suggested minimal commands:

```bash
touch README.md .gitignore .editorconfig
mkdir -p docs docs/adr .github/workflows
```

```bash
go mod init github.com/<your-user-or-org>/coralhub-backend
```

Use the backend-specific module path from the start.

---

## 5. First Local Commit

Stage initial files:

```bash
git add .
```

Create the first commit:

```bash
git commit -m "chore(repo): bootstrap CoralHub backend repository"
```

Do not wait to create the first commit until the whole architecture exists.

The first commit should establish the repository as a real working project.

---

## 6. Create the Remote Repository on GitHub

Create a new GitHub repository with:

- name: `coralhub-backend`
- visibility: `private` unless you explicitly want it public
- do not add autogenerated README, `.gitignore`, or license if you already created files locally

If using GitHub CLI:

```bash
gh repo create coralhub-backend --private --source=. --remote=origin --push
```

If creating manually in the GitHub UI:

1. Create the repository.
2. Copy the remote URL.
3. Add the remote locally:

```bash
git remote add origin git@github.com:<your-user-or-org>/coralhub-backend.git
```

4. Push:

```bash
git push -u origin main
```

## 7. Recommended Initial GitHub Settings

After the remote exists, configure:

- default branch: `main`
- branch protection on `main`
- pull request required before merge
- require green CI before merge
- require at least one review
- block force push to `main`
- block direct push to `main`
- enable squash merge

Optional but recommended:

- CODEOWNERS
- issue templates
- PR template
- Dependabot alerts
- secret scanning

---

## 8. Recommended Branch Strategy

Use trunk-based development with short-lived branches.

Branch naming:

- `feat/<short-description>`
- `fix/<short-description>`
- `refactor/<short-description>`
- `chore/<short-description>`
- `docs/<short-description>`
- `ci/<short-description>`
- `infra/<short-description>`

Examples:

- `feat/bootstrap-api-and-worker`
- `feat/add-tenant-bootstrap`
- `feat/choirs-vertical-slice`
- `feat/events-and-reminders`
- `infra/add-ecs-service-definitions`
- `ci/add-go-and-sqlc-checks`

---

## 9. Recommended Commit Strategy

Use Conventional Commits.

Examples:

- `chore(repo): bootstrap CoralHub backend repository`
- `feat(tenants): seed Coral Jovem Asa Norte tenant`
- `feat(choirs): add create and list flows`
- `feat(events): schedule reminders on event creation`
- `feat(files): add S3 upload support`
- `infra(aws): add ECS Fargate service definitions`
- `ci(actions): add test and lint workflow`

Keep commits focused.

Do not combine:

- infra changes
- feature changes
- wide refactor
- formatting noise

in one unclear commit.

## 11. Recommended First Milestones After Repo Creation

After the repository is live, implement in this order:

1. repository and CI bootstrap
2. Go project bootstrap
3. tenant and tenant-config bootstrap
4. API and worker entrypoints
5. config, logger, health check
6. first business vertical slice: choirs and users

Do not spend the first days only creating folders and abstractions.

---

## 12. What the AI Should Do vs What You Should Do

### Human owner

You should do:

- create the GitHub repo
- define visibility
- configure branch protection
- manage org/repo permissions
- configure GitHub secrets
- approve infrastructure ownership decisions

### AI

The AI should do:

- generate project files
- generate docs and ADRs
- scaffold the codebase
- implement features
- create CI files
- create Terraform or infra code

The AI can suggest repository settings, but ownership stays with you.
